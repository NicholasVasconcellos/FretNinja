# Task 019: Signal Smoothing, Edge Cases & Latency Validation

## Context
Tasks 014–018 delivered a working end-to-end pitch detection pipeline. This task hardens it: smooth noisy output, handle edge cases, and measure actual latency to confirm <15ms target.

## Steps
1. Add an EMA (Exponential Moving Average) filter in the C++ layer:
   - Smooth frequency output to reduce jitter between consecutive detections.
   - Configurable smoothing factor (alpha ≈ 0.3–0.5).
   - Reset the filter on `stop()` or when confidence drops below threshold (avoids stale smoothing).
2. Handle edge cases in the DSP path:
   - Silence: return frequency=0, confidence=0 immediately (skip YIN if RMS below threshold ~0.01).
   - Clipping: detect if signal is clipping (samples near ±1.0), flag in result or log warning.
   - Frequency out of range: clamp or return 0 if detected pitch is outside E2–C8.
   - Rapid note changes: reset EMA when detected frequency jumps more than 1 semitone between frames.
3. Add a pre-filter in C++:
   - Optional simple high-pass filter (~60Hz cutoff) to reject DC offset and low-frequency rumble.
4. Measure and log end-to-end latency:
   - Timestamp when audio buffer arrives in native callback.
   - Timestamp when `getLatestPitch()` is called from JS.
   - Log the delta. Target: <15ms from buffer arrival to JS retrieval.
   - Add a `getLatencyMs()` method to the TurboModule for debugging.
5. Update the `usePitchDetector` hook to expose an optional `debug` mode that includes latency info in the returned object.

## Acceptance Criteria
- [ ] Frequency output is stable (no jitter >5 cents) on a sustained tone
- [ ] Silence produces null/zero pitch consistently, no phantom detections
- [ ] Rapid note changes (staccato) are detected within 2 frames (~46ms)
- [ ] Measured latency from audio callback to JS retrieval is <15ms on a mid-tier device
- [ ] High-pass filter removes DC offset without affecting guitar fundamentals
- [ ] EMA resets properly on stop/start and large pitch jumps
