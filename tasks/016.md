# Task 016: iOS Audio Capture with AVAudioEngine

## Context
Tasks 014 (module scaffold) and 015 (C++ DSP) are complete. This task wires up real microphone input on iOS using AVAudioEngine, feeds audio into the C++ YIN engine, and stores results for JSI retrieval.

**Important — audio session coexistence:** The app uses `expo-audio` for sound effects. `utils/sounds.ts` calls `setAudioModeAsync({ allowsRecording: true })` which sets the AVAudioSession category to `.playAndRecord`. The new module's AVAudioEngine must coexist with this — do not reconfigure the session category if it is already `.playAndRecord`. See commit `5d6a9f9` for context on why `loadSounds()` must be called before `pitch.start()`.

## Steps
1. In the iOS native module, set up AVAudioSession:
   - Category: `.playAndRecord` with `.defaultToSpeaker` and `.allowBluetooth` options.
   - Request microphone permission at runtime via `AVAudioSession.requestRecordPermission`.
   - Check the current session category before reconfiguring — if already `.playAndRecord`, skip reconfiguration to avoid conflicts with `expo-audio`.
2. Create an `AudioCaptureManager` (Swift class) that:
   - Initializes `AVAudioEngine` and installs a tap on `inputNode`.
   - Tap format: float32 PCM, 44100Hz, mono.
   - On each tap callback, writes samples into the C++ ring buffer.
3. Bridge Swift ↔ C++:
   - Use an ObjC++ (.mm) bridging file if needed, or use Swift's C interop.
   - The bridge should call `YIN::detect()` and `NoteMapper::mapToNote()` when enough samples are in the ring buffer.
   - Store the latest `PitchResult` + `NoteInfo` in a thread-safe shared location (atomic struct or mutex-guarded).
4. Wire up the TurboModule methods:
   - `start()`: request mic permission → configure audio session → start AVAudioEngine → return Promise.
   - `stop()`: remove tap → stop engine → release resources.
   - `getLatestPitch()`: synchronous read of the latest stored result, return as JS object.
5. Handle audio interruptions (phone calls, Siri) — pause/resume gracefully.
6. Ensure the audio tap callback does NOT allocate memory or block.
7. Verify that starting the new pitch detector does not break `expo-audio` sound playback. Both must coexist since the quiz plays SFX while detecting pitch.

## Acceptance Criteria
- [ ] `start()` requests mic permission and begins audio capture on iOS
- [ ] `stop()` cleanly shuts down the audio engine with no leaks
- [ ] `getLatestPitch()` returns real pitch data when a tone is played into the mic
- [ ] Audio callback runs on high-priority thread with no allocations
- [ ] App handles audio interruptions without crashing
- [ ] Builds successfully via `npx expo prebuild --platform ios && xcodebuild`
- [ ] `expo-audio` sound playback still works while pitch detection is active
